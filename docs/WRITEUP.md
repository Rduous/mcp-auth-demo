# MCP Authorization — Design Decisions, Agent Interactions, and a POV

## Design Decisions

**Language** I chose Python due to its well-supported / easy to use libraries around HTTP servers and MCP. I don't regret it!

**I built on the official MCP SDK's own authorization support rather than hand-rolling the protocol mechanics, and that was a deliberate speed-for-depth tradeoff.** The SDK already ships a working RFC 9728 `401` + Protected Resource Metadata response server-side (`TokenVerifier`/`AuthSettings`), and a client-side `OAuthClientProvider` that handles AS discovery, CIMD-based `client_id`, PKCE, and even the RFC 6750 §3.1 `insufficient_scope` step-up retry once the server's `WWW-Authenticate` header carries the right `scope=` field. Leaning on it got a spec-correct, end-to-end proof of concept running fast, without re-deriving well-established RFC mechanics from scratch just to prove the pattern works. The real cost: it means a meaningful slice of "the protocol" in this repo is SDK plumbing rather than code I wrote by hand, which understates how much I actually dug into the underlying RFC mechanics along the way (run `python3 client/main.py get-logs --topic "step-up"` for one concrete example — reading the SDK's own source to understand exactly why a missing header field was silently breaking step-up). If an evaluator wants the discovery/PKCE/step-up logic implemented by hand instead, that's a bounded, backable-out substitution: the SDK's role is scoped to well-defined interfaces (`TokenVerifier.verify_token`, the `OAuthClientProvider` constructor), and none of the actual authorization *logic* — introspection-based verification, audience binding, per-tool scope enforcement — lives inside the SDK; that's already separate, hand-written code in `server/auth.py` and `server/scope_gate.py`.

**Choosing the authorization approach was the biggest fork in the road, and I treated it that way.** I tackled this question before writing a line of code, landing on Authlete (an SaaS offering).

I ruled out mature open-source options (Ory Hydra) early in the process, after confirming they did not yet support CIMD. (Ory Hydra has an open GitHub feature request about it, not a shipped capability.) I asked Claude for its top SaaS recommendations. Before finalizing my choice, I ran a manual, curl-driven spike against candidate authorization servers rather than trusting vendor documentation. This mattered: CIMD (Client ID Metadata Documents) is a very recent addition to the MCP authorization spec, and support for it turns out to be unevenly distributed even among providers that market themselves as MCP-ready.  I settled on Authlete over WorkOS despite WorkOS having a more polished, fully-hosted authorization flow, because Authlete's CIMD support was verifiably real (evidenced by a versioned service flag with dev-mode controls) and I'd already validated it end-to-end before committing.


**That same spike surfaced an architectural gap I hadn't anticipated: Authlete is a backend API, not a hosted authorization server.** It has no live login/consent page — `/authorize` and `/token` don't exist as real HTTP endpoints anywhere; a developer is expected to build a thin frontend that calls Authlete's business logic API. I confirmed this concretely (a 404 at Authlete's own well-known discovery endpoint) rather than assuming it, and decided to build that frontend myself rather than switch providers, since I'd already validated the parts of Authlete that mattered (CIMD, PKCE enforcement, audience binding) and the missing piece was small and well-scoped. I kept sign-in as an explicit, labeled no-op (auto-approve a single demo subject) rather than pretending to solve identity — real identity (Google SSO plus an allow-list) is scoped as a deliberate future refinement, not something I hand-waved into the current implementation.

Choosing Authlete shaped the implementation, requiring me to stand up an additional auth service. While it's possible that WorkOS (a fully-hosted auth service) might have worked with one fewer servers, I'm happy with my choice: the custom login page is a great place to hard-code testing scenarios, like "Log in in a way that's expected to fail."

**I chose not to trust the authorization server's own audience check, and built the resource server to verify it independently.** Testing revealed that Authlete's introspection API only enforces resource matching when explicitly asked to check it alongside a required scope in the same call. Called the way our resource server actually calls it — a token with no specific scope to check yet — it reports the token's bound resources but does not reject a mismatch on its own. Rather than depend on a call pattern that only sometimes gets Authlete's native enforcement, I built the resource server to perform that check independently, every time, regardless of how it's called.

**Per-tool scope enforcement needed its own design decision, and I made the call based on the assignment's actual wording, not just convenience.** MCP's client-server relationship is 1:1 per session; a host application can juggle multiple server connections, but that's a layer above what we were asked to build. Given the assignment described "an MCP server" and "your protected MCP tool" in the singular, I concluded the intended shape was one server exposing multiple tools, with at least one tool scope-gated — not scope tiers split across separate servers. I built custom middleware that inspects each tool call independently, over the simpler option of running separate FastMCP instances per scope tier, specifically so the step-up story (call a tool, get `403`, re-authenticate narrowly, retry) stays inside one continuous session, matching how a real deployment (a Claude-based host on a user's laptop, talking to one remote server) would actually behave.

**An "extended log" tool to support the writeup and exercise scope-gating.** I decided to add a second tool to fully explore scoping, and I decided to make it "useful." The logs tool can print out excerpted interactions between me and the agent, to give interested readers more insight without bloating the writeup. One honest caveat: the content it returns is pulled directly from NOTES.md, a file that already lives in this public repo's git history — so the scope gate isn't providing real confidentiality; anyone willing to read the repo's commits can already see everything the tool returns. The point of gating it behind logs:read is to demonstrate the access-control pattern itself — a second scope with a genuinely different permission boundary, enforced the same way as the primary tool — not to hide anything. I considered and rejected returning a link to externally-hosted content instead, since a bare link has no ongoing enforcement once issued; keeping the content in-repo and serving it in-band from the resource server keeps the enforcement meaningful, even though the underlying data was never actually secret.

**Hosting: Docker containers on Render's free tier, with one caveat worth knowing before you test.** Both services (`server`, `authserver`) run as Docker containers deployed via a Render Blueprint (`render.yaml`, checked into this repo) — real, always-reachable HTTPS URLs, at no cost, so evaluation doesn't depend on me being present to run anything locally. The one tradeoff that comes with the free tier: Render spins a service down after about 15 minutes of inactivity, and the next request after that wakes it back up — typically under a minute, but that first hit can time out or feel unusually slow rather than failing cleanly. If the very first call you make hits an odd error or a long hang, that's most likely a cold start, not a broken deployment; a retry, or waiting roughly a minute before trying again, should resolve it.

**Known gap: silent refresh-token behavior is requested but not independently verified.** `client_metadata` requests `refresh_token` as a grant type ([client/main.py:194](../client/main.py)), and the SDK's `OAuthClientProvider` is capable of using it to renew an expiring access token without a browser popup. But I never built or ran a scenario that isolates and confirms this specifically — the one place it came up ([docs/NOTES.md](NOTES.md), the entry on `probe`'s origin) was as a risk to route *around*: testing expiration deliberately uses `probe`, which sends a static bearer header instead of going through `OAuthClientProvider`, specifically so the SDK's own possible silent refresh wouldn't mask the token having actually expired. That was the right call for testing expiration cleanly, but it means refresh-on-expiry itself — does a plain `get-time` call succeed silently after expiry, using the stored refresh token, with no re-auth? — is a real, live question I haven't answered, not something I've confirmed works.

## Interactions With the Agent

I used Claude as an implementation and research partner, not as an autopilot — I set the direction and the standards for verification; Claude wrote the majority of the code, ran the manual API tests, and did most of the SDK/documentation research.

**Where I directed rather than just reacted:** I pushed back multiple times when Claude's conclusions weren't actually verified. When Claude reported that Authlete's introspection API "doesn't enforce audience matching," I asked directly why — not just accepting that it was true, but asking for the mechanism. That question led to discovering the actual root cause: every test token we'd minted had a null scope, because we'd never registered our custom scopes with Authlete, which made the whole test uninterpretable. I insisted on a properly isolated re-test rather than a second guess, which produced the real, confirmed answer (run `python3 client/main.py get-logs --topic "resource check does exist"` for that specific account — "correction" alone also matches an unrelated later entry about CIMD's scope field, since the match is a substring search over section headings). I also caught Claude skipping a question I'd asked several turns earlier, redirected it away from open-ended research back toward incremental, checked-in progress when I felt the pace of exploration outweighed the value it was adding, and set the collaboration style itself — asking for smaller, more explained steps at points where I wanted to understand the mechanics rather than just receive a working diff.

**Where Claude did the heavy lifting:** writing essentially all of the Python (the resource server, the client, the thin authorization-server frontend, the scope-enforcement middleware), running the curl-based manual testing against Authlete's API during the de-risking phase (run `python3 client/main.py get-logs --topic "phase 0"` for the full account), reading the MCP Python SDK's source directly to confirm exact behavior instead of trusting summarized documentation, and debugging from raw server logs when something didn't work as expected (a stale token, an unregistered scope, a missing header field — run `python3 client/main.py get-logs --topic "step-up"` for a good example of this pattern end to end).

**Net assessment:** this was a genuine directed-agent workflow. I made the calls on ambiguous points (which authorization server, how to structure per-tool scope enforcement, how to avoid the shareable-link access-control trap), caught real mistakes before they became wrong conclusions in this write-up, and pushed for verification over assumption repeatedly. Claude supplied the implementation speed and the systematic testing discipline once a direction was set.

## POV: When Does This Pattern Make Sense in the Real World?

**This pattern exists to solve a real N×M problem.** Without a shared standard, every combination of agent (Claude, Gemini, ChatGPT, and all their various deployments) and tool would need its own bespoke integration — its own credential format, its own consent flow, its own trust model. MCP solves that for tool integration generally; this authorization profile solves the same N×M problem specifically for *access control*. Without it, every agent-tool pairing would need its own custom auth handshake — exactly the kind of bespoke integration cost the base protocol exists to eliminate in the first place. Each agent implements the standard once, each tool implements the standard once, and access control "just works" between any pairing of the two.

**It's also specifically designed to keep credentials away from the model itself.** An agent never needs to see or hold a user's actual password or long-lived API key — it gets a scoped, time-limited bearer token through a flow it doesn't control the internals of. That distinction matters more with agentic systems than with traditional apps: a compromised or misbehaving agent leaking a scoped, short-lived, revocable token is a very different blast radius than one leaking a raw credential. This repo demonstrates the actual mechanics behind that claim, not just the assertion — PKCE protects the handshake itself from interception, audience binding means a token issued for one tool can't be replayed against another (the "confused deputy" problem), and scope enforcement means even a legitimately-issued token only grants a specific slice of capability.

**Where I think this pattern gets more expensive than it looks: the resource server's validation strategy becomes a real cost decision at scale, not a footnote.** The spec doesn't mandate a token format — opaque tokens validated via introspection, or self-contained JWTs validated locally. I hit this tradeoff directly in this project: I chose JWTs specifically to keep the *option* of skipping the introspection network hop, and weighed that explicitly against giving up live revocation checking. At real scale — tens of thousands of users generating tens of thousands of requests — that stops being an abstract tradeoff. Introspection-per-request against a metered SaaS authorization server (the choice we made here) means every protected request costs real money and adds real latency. A JWT-validated deployment scales for free, but only by accepting that a revoked token stays valid until it expires. That's a genuine architectural decision with a real cost curve behind it, not a detail to defer.

**It also doesn't solve role provisioning, and I don't think it's supposed to.** CIMD's entire trust model is domain ownership — you control this URL, therefore the authorization server trusts your claimed identity. That's a good fit for the open scenario MCP was built for: arbitrary agents talking to arbitrary tool providers across the internet, with no pre-existing relationship. But it has nothing to say about "should this identity be allowed to have this scope in the first place" — answering that requires exactly the kind of centralized registration and gatekeeping CIMD exists to avoid. A large enterprise wanting audited, revocable, role-based access would likely want Dynamic Client Registration behind an approval workflow, or a private client allowlist, instead of CIMD's fully open model. That's not a flaw in CIMD — it's optimized for a different trust problem than the one most enterprises have internally, and reaching for it inside a corporate boundary would be solving the wrong problem with the right-looking tool.

**Net POV:** this pattern earns its complexity when you actually have the N×M problem it was built for — many agents, many tools, no pre-existing trust relationship between them, and real security stakes if a credential leaks. It's a worse fit — or at least an incomplete one — inside a single organization with a small, known set of internal tools, where the harder problem is upstream (who gets which role) rather than downstream (how a token gets validated once issued), and where introspection-per-call economics can make the "just use a SaaS authorization server" default genuinely expensive at scale.
