# MCP Authorization — Design Decisions, Agent Interactions, and a POV

**A note to evaluators on time spent:** I spent more time on this assignment than I was supposed to. I usually work with Gemini, and I got genuinely absorbed exploring what Claude could do. I had fun and didn't want to stop! Thanks for bearing with me here.

## Design Decisions

**Language:** I chose Python due to its well-supported / easy to use libraries around HTTP servers and MCP. I don't regret it!

**Two protected tools.** `get_time` just returns the current UTC time — deliberately trivial, since the real signal here is the auth mechanics, not tool logic. `get_logs`, gated behind a different scope (`logs:read`), returns excerpts from this project's own work log (`NOTES.md`), with an optional `--topic` filter — real, useful content, not a placeholder, so the second scope actually gates something worth reading. That log is already public in this repo's git history, so `logs:read` isn't providing real confidentiality; the point is demonstrating the access-control pattern itself — a second scope with a genuinely different permission boundary, enforced the same way as the primary tool. Sample commands pulling specific topics out of it show up throughout the rest of this write-up rather than being repeated here.

**I built on the MCP SDK's own auth support (`TokenVerifier`/`AuthSettings`, `OAuthClientProvider`) rather than hand-rolling discovery, PKCE, and step-up.** That got the prototype up fast, but didn't mean skipping the protocol itself — e.g. `python3 client/main.py get-logs --topic "came almost for free"` covers a case where fixing a bug meant reading the SDK's source down to a missing header field. It's also an easy swap later, if requested by reviewers; and plenty of real authorization logic — introspection, audience binding, per-tool scope enforcement — is already separate, hand-written code in `server/auth.py` and `server/scope_gate.py`.

**Choosing the authorization server was the biggest fork in the road.** I ruled out leading OSS choice Ory Hydra early — CIMD support there is an open GitHub issue, not shipped. CIMD is recent enough that support is uneven even among self-described MCP-ready SaaS providers, so I tested CIMD and token workflows via curl rather than trusting docs (`python3 client/main.py get-logs --topic "curl spike"`). I picked Authlete because its CIMD support was the most verifiably real (a versioned service flag with dev-mode controls), validated end-to-end before committing.

**My choice of Authlete required building an auth server.** Authlete is a backend API, not a hosted AS — no live `/authorize` or `/token`; you build a thin frontend against its business logic API (`python3 client/main.py get-logs --topic "backend-API-only"`). I wanted a login page, so the choice was switch to a fully-hosted option like WorkOS, or build that frontend myself. I built it, and I'm glad I did: it saved re-testing a second provider, and the custom login page turned out to be a great place to hard-code test scenarios like "log in in a way that's expected to fail."

**User identification on that frontend is a deliberate no-op, not an oversight.** `/authorize` auto-approves one hardcoded `DEMO_SUBJECT` rather than asking for a username and password — the "sign in" links only ever choose *scope*, not *who*. I judged it the least protocol-relevant piece to build for this assignment: everything that actually gates access — `server/auth.py`'s audience check, `scope_gate.py`'s per-tool scope check — runs on the token's scopes and resource, never on identity, so a stubbed subject doesn't weaken anything this project is meant to demonstrate. Real identity (Google SSO + an allow-list) is deferred as Phase 8 and slots into the same `subject` field already passed to Authlete's `/auth/authorization/issue` — see the client-identity-vs-user-identity discussion in the POV section for why that's a different problem, not just unfinished work.

**I don't trust the AS's own audience check — the resource server verifies it independently, and this isn't just theoretical.** Tracing every code path: `initialize` and `tools/list` never reach a per-tool scope check at all, so the resource server's own manual audience check is the *only* enforcement that runs. Even for a scope-gated tool call, the resource check happens first — Authlete's own combined scopes+resources check only ever gets trusted for a genuine scope verdict once the resource is confirmed to match ourselves. This isn't hypothetical: a wrong-resource token with an otherwise-valid scope briefly produced a misleading `403 insufficient_scope` instead of `401` when that ordering slipped (`check_scope` treated Authlete's `FORBIDDEN` as always meaning insufficient scope, when it doesn't distinguish the two) — caught by re-running the test suite, not code review, and fixed by checking resource before trusting any scope verdict (`python3 client/main.py get-logs --topic "mislabeled"`).

**Two tools, one MCP server — not scope tiers split across separate servers.** The MCP SDK forced this decision, not the spec: FastMCP's `streamable_http_app()` wraps the entire session in one route behind a single flat `required_scopes` list, with no native per-tool granularity. Rather than run a separate FastMCP instance per scope tier — matching the SDK's own structure, but splitting `get_time` and `get_logs` across two servers — I wrote middleware that inspects each tool call independently and checks that specific tool's required scope (`python3 client/main.py get-logs --topic "custom middleware"`). That keeps step-up (call a tool, get `403`, re-authenticate narrowly, retry) inside one continuous session, and it happens to match the assignment's own wording too ("an MCP server," "your protected MCP tool," singular).

**Hosting: Docker on Render's free tier, with one caveat.** Both services deploy via a Render Blueprint (`render.yaml`) to real, always-on HTTPS URLs (`python3 client/main.py get-logs --topic "containerization"`), so evaluation doesn't depend on me running anything locally. **NOTE:** Free-tier services sleep after ~15 minutes idle and take under a minute to wake — a slow or timed-out first request is a cold start, not a broken deployment; retry.

**Known gap: refresh-token behavior is requested but not verified.** `client_metadata` requests `refresh_token` as a grant type ([client/main.py:194](../client/main.py)), and `OAuthClientProvider` can use it to silently renew an expiring token — but I never isolated a scenario confirming it. Testing expiration deliberately uses `probe` (a static bearer header, bypassing the SDK) so silent refresh wouldn't mask real expiry (`python3 client/main.py get-logs --topic "persistent token storage"`) — the right call for testing expiration cleanly, but it leaves refresh-on-expiry itself unverified.

## Interactions With the Agent

1. **First I explored the problem with the agent** — working through the MCP auth spec and the assignment together, checking my understanding by paraphrasing it back and inviting correction.
2. **Then I developed a plan:** I supplied the real constraints (a simple language for lightweight web servers; "one command" meaning a scripted CLI) and used the agent to turn my decisions into a structured `PLAN.md`.
3. **Then we bootstrapped the project together** — I set guardrails ("keep it simple," "let's be rigorous") and caught wrong turns as they happened: challenging the agent's diagnosis with log evidence instead of taking it at face value, and insisting on an isolated re-test over a second guess.
4. **Then we planned deployment** — I re-derived the actual requirement (grader access without me present, cost) instead of accepting an impressive-looking AWS/Terraform plan, which turned a $20-30/mo build into a free Docker+Render one.
5. **Finally, agent-driven testing** — I pushed to actually run all eight scripted scenarios rather than leave them as documentation, which surfaced three real, live bugs (a consent-driver deadlock, a production 404, revocation silently no-op'ing) (`python3 client/main.py get-logs --topic "found three real bugs"`). A ninth scenario, added after a friend's clean install hit a cross-process step-up failure, caught a fourth (`python3 client/main.py get-logs --topic "cross-process"`).

When I work with an agent, I use these principles:

- **Maintain project context via version-controlled artifacts** — `PLAN.md`, `NOTES.md`, and similar checked-in docs, not conversational memory, are the source of truth.
- **Treat claims as hypotheses, not facts** — including my own. When something "doesn't exist" or "should work," ask for the mechanism before accepting it.
- **Paraphrase before code exists** — restate a plan or design back and invite correction, since a misunderstanding caught before implementation is nearly free.
- **Settle disagreements with evidence, not seniority** — a wrong diagnosis, mine or the agent's, gets resolved by a log line or test output, not by who proposed it first.
- **Gate risk explicitly** — separate authorization for low-risk vs. high-risk actions (commit vs. push), keep secrets out of shared context, draw a clear line on who edits a file mid-flight.
- **Verify empirically rather than trust a status report** — run it myself, capture raw output, and spot-check a checked box before building on it.

The result is a project shaped by my own understanding and judgment, not the agent's defaults. It's a solid POC, verified by a test suite that's easy to run and makes regression detection straightforward. I hope it also gives reviewers real insight into what they're evaluating, 

My progress with the agent wasn't monotonic. I explore the problem space at the same time as I ask the agent to write code, rather than fully resolving direction first — which sometimes produces wrong turns, not just clean forward motion. The clearest example: I had the agent write a detailed AWS/Terraform plan (Fargate vs. EKS, ALB routing, TLS) before the real constraint — grader access without me present, cost — had surfaced, and most of it got discarded in favor of Docker+Render. That's real token spend that didn't survive. But I think the exploratory style is worth it: it's how I actually learn the material, and the wrong AWS-first instinct only became visible because I'd built it out far enough to compare against the real constraint once it emerged. Messier than a straight line doesn't mean less directed — the direction-finding happened partly through building, not only before it.

**Full Disclosure: A humbling comparison.** I had a separate Claude agent tackle this assignment on a fast setting. With very little input it produced a working version of this same assignment — its own AS, its own JWT/JWKS validation, real audience binding and scope enforcement, a headless e2e test — in about 17 minutes and 45 tool calls. This project took multiple sessions, and I don't want to know the token or wall-clock multiple! Some of that gap bought real things theirs doesn't have (a live deployment, step-up auth, actual revocation, four bugs no one would have caught by reading code); some of it just bought me time to argue with the agent, second-guess it, and learn the spec by doing it the slow way.

## POV: When Does This Pattern Make Sense in the Real World?

**This pattern exists to solve a real N×M problem.** Without a shared standard, every agent (Claude, Gemini, ChatGPT, and their many deployments) paired with every tool would need its own bespoke credential format, consent flow, and trust model. MCP solves that for tool integration generally; this authorization profile solves the same N×M problem for *access control* specifically. Each agent implements the standard once, each tool implements it once, and access control just works between any pairing of the two.

**It's also designed to keep credentials away from the model itself.** An agent never sees a user's actual password or long-lived API key — it gets a scoped, time-limited bearer token instead. That matters more for agentic systems than traditional apps: a misbehaving agent leaking a scoped, revocable token is a very different blast radius than leaking a raw credential. This repo demonstrates the mechanics, not just the claim — PKCE protects the handshake from interception (`python3 client/main.py get-logs --topic "PKCE now enforced"`), audience binding stops a token issued for one tool being replayed against another (the "confused deputy" problem) (`python3 client/main.py get-logs --topic "audience rejection proven"`), and scope enforcement caps even a legitimately-issued token to a specific slice of capability.

**Where this pattern gets more expensive than it looks: token-validation strategy is a real cost decision, not a footnote.** The spec doesn't mandate a format — opaque tokens validated via introspection, or self-contained JWTs validated locally (`python3 client/main.py get-logs --topic "Switched to JWT"`). Some may choose JWTs specifically to have the option of skipping the introspection round-trip, but this means giving up live revocation checking. At real scale, what you choose has a big impact: introspection-per-request against a metered SaaS AS (what we built) costs real money and latency on every protected request. JWT validation scales for ~free but accepts that a revoked token stays valid until it expires. That's a genuine architectural tradeoff, not a detail to defer.

**It also solves client identity, not user identity — a different, harder problem.** Inside a large org, the hard question isn't "how do I implement OAuth" — it's "should this *user* be allowed to do this, and how do I audit that over time." CIMD only ever answers "which piece of software is calling," proven by domain ownership, not "which human is behind it or what they should be allowed to do" — this project keeps that distinction explicit by treating real user sign-in (Google SSO, an allow-list) as a separate, deferred piece ([Phase 8](PLAN.md#phase-8)). A more gated alternative, like Dynamic Client Registration behind approval, doesn't close that gap either — it just changes how client applications get vetted, the same category of problem as CIMD, with a human approving instead of domain ownership proving. Neither touches which roles should hold which permissions; that's an identity-and-access-governance layer above any client-identification mechanism, CIMD included. Not a flaw in CIMD — it was never scoped to solve that problem.

**Net POV:** this pattern earns its complexity when you actually have the N×M problem it solves — many agents, many tools, no pre-existing trust, real stakes if a credential leaks. It's a worse fit inside a single org with a small, known set of internal tools, where the harder problem is upstream (who gets which role) rather than downstream (validating a token), and where introspection-per-call economics can make "just use a SaaS AS" expensive at scale.
