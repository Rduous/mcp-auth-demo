# MCP Authorization — Design Decisions, Agent Interactions, and a POV

## Design Decisions

**Language** I chose Python due to its well-supported / easy to use libraries around HTTP servers and MCP. I don't regret it!

**I built on the official MCP SDK's own authorization support (`TokenVerifier`/`AuthSettings` server-side, `OAuthClientProvider` client-side) rather than hand-rolling discovery, PKCE, and step-up retry.** Using SDK methods doesn't mean I didn't get hands-on with the protocol — e.g. run `python3 client/main.py get-logs --topic "step-up"` for a case where understanding *why* step-up was silently breaking meant reading the SDK's own source down to a missing header field. I chose to do this to get the prototype up fast, but it's an easy substitution if an evaluator wants it: the SDK's role is scoped to well-defined interfaces (`TokenVerifier.verify_token`, the `OAuthClientProvider` constructor), while the actual authorization logic — introspection-based verification, audience binding, per-tool scope enforcement — is already separate, hand-written code in `server/auth.py` and `server/scope_gate.py`.

**Choosing the authorization server was the biggest fork in the road.** I ruled out mature open-source options (Ory Hydra) early — CIMD support there is an open GitHub issue, not a shipped capability. CIMD is a very recent spec addition, so support is unevenly distributed even among self-described MCP-ready providers; when evaluating SaaS providers, I tested CIMD and token workflows via curl rather than trusting vendor docs. I picked Authlete because it had the most verifiably-real CIMD support (a versioned service flag with dev-mode controls), and I'd already validated it end-to-end before committing.

**My choice of Authlete required an additional component: an auth server.** Authlete is a backend API, not a hosted authorization server. It has no live login/consent page — `/authorize` and `/token` don't exist as real HTTP endpoints anywhere; a developer is expected to build a thin frontend that calls Authlete's business logic API. I wanted a login page, so I had a choice: switch to a fully-hosted solution like WorkOS, or build a thin auth service. I chose the latter, and I'm happy I did: I saved time testing a second auth provider, plus the custom login page is a great place to hard-code testing scenarios, like "Log in in a way that's expected to fail."

**I don't trust the authorization server's own audience check — the resource server verifies it independently.** Authlete's introspection API only enforces resource matching when a scope check is requested in the same call; called the way our resource server actually calls it, it reports the token's bound resources but doesn't reject a mismatch on its own. Rather than depend on that inconsistency, the resource server checks audience itself, every time.

**Per-tool scope enforcement: one server exposing multiple tools, not scope tiers split across separate servers.** MCP's client-server relationship is 1:1 per session, and the assignment described "an MCP server" and "your protected MCP tool" in the singular — so I built middleware that inspects each tool call independently, rather than running separate FastMCP instances per scope tier. That keeps the step-up story (call a tool, get `403`, re-authenticate narrowly, retry) inside one continuous session, matching how a real deployment would actually behave.

**An "extended log" tool exercises scope-gating with real content.** It returns excerpts of `NOTES.md`, which already lives in this public repo's git history — so `logs:read` isn't providing real confidentiality; the point is to demonstrate the access-control pattern itself, a second scope with a genuinely different permission boundary enforced the same way as the primary tool.

**Hosting: Docker containers on Render's free tier, with one caveat worth knowing before you test.** Both services deploy via a Render Blueprint (`render.yaml`) to real, always-on HTTPS URLs, so evaluation doesn't depend on me running anything locally. One tradeoff: free-tier services sleep after ~15 minutes idle and take under a minute to wake — if your first request times out or hangs, that's a cold start, not a broken deployment; retry.

**Known gap: refresh-token behavior is requested but not independently verified.** `client_metadata` requests `refresh_token` as a grant type ([client/main.py:194](../client/main.py)), and the SDK's `OAuthClientProvider` can use it to silently renew an expiring token. I never isolated a scenario confirming this — testing expiration deliberately uses `probe` (a static bearer header, bypassing the SDK) specifically so silent refresh wouldn't mask real expiry ([docs/NOTES.md](NOTES.md)). Right call for testing expiration cleanly, but it means refresh-on-expiry itself remains a real, unanswered question.

## Interactions With the Agent

1. **First I explored the problem with the agent** before writing anything — working through the MCP auth spec and the shape of the assignment together, checking my own understanding by paraphrasing it back and inviting correction rather than assuming I had it right.
2. **Then I developed a plan:** I supplied the actual constraints myself (a language that gives simple, lightweight web servers; "in one command" meaning a scripted CLI) and used the agent to turn my own decisions into a structured `PLAN.md`, not to make those decisions for me.
3. **Then we bootstrapped the project together** — I set concrete guardrails ("keep the code simple," "let's be rigorous") and caught wrong turns as they happened: challenging the agent's diagnosis with actual log evidence instead of taking its theory at face value, and insisting on a properly isolated re-test rather than a second guess.
4. **Then we planned deployment** — I re-derived the actual requirement (grader access without me present, cost) instead of accepting a more impressive-looking AWS/Terraform plan at face value, which is what turned a $20-30/mo build into a free Docker+Render one.
5. **Finally we moved into agent-driven testing** — I pushed to actually run the eight scripted scenarios for real rather than leave them as documentation, which is what surfaced three genuine, live bugs (a consent-driver deadlock, a production 404, revocation silently no-op'ing).

When I work with an agent, I use these principles:

- **Treat claims as hypotheses, not facts** — including my own. When something "doesn't exist" or "should work," ask for the mechanism before accepting it.
- **Paraphrase before code exists** — restate a plan or design back and invite correction, since a misunderstanding caught before implementation is nearly free.
- **Settle disagreements with evidence, not seniority** — a wrong diagnosis, mine or the agent's, gets resolved by a log line or test output, not by who proposed it first.
- **Gate risk explicitly** — separate authorization for low-risk vs. high-risk actions (commit vs. push), keep secrets out of shared context, draw a clear line on who edits a file mid-flight.
- **Verify empirically rather than trust a status report** — run it myself, capture raw output, treat a checked box as something to spot-check, not something to build on unexamined.

The outcome of those behaviors is a project that reflects my own understanding of the assignment and has been shaped throughout by my technical judgment, not the agent's defaults — down to specific choices like Python (a constraint I stated, not a suggestion I accepted), the client defaulting to deployed services over localhost (a reversal of my own first design), and the `MCP_AUTH_CONSENT_RETRY` variable (my own reasoning about what a retry scenario needed). I also hope it gives reviewers real insight into what they're evaluating, backed by a test suite that's easy to execute and makes verification and regression detection straightforward, rather than something that only looks correct on paper.

My progress through this wasn't monotonic. I explore the problem space at the same time as I'm asking the agent to write code, rather than fully resolving direction first and then executing cleanly — and that sometimes produces wrong turns or miscommunications, not just clean forward motion. The clearest example: I had the agent write a genuinely detailed AWS/Terraform deployment plan (Fargate vs. EKS, ALB routing, TLS termination) before the real constraint — grader access without me present, cost — had actually surfaced, and most of that plan got discarded a few turns later in favor of Docker+Render. That's real token spend that didn't survive into the final project. But I think the hands-on, exploratory style is worth that cost: it's how I actually learn the material, and it's also what caught things a cleaner, fully-pre-planned process would have missed — the wrong AWS-first instinct only became visible *because* I'd asked for it built out far enough to compare against the real constraint once it emerged. The interaction being messier than a straight line doesn't mean it was less directed; it means the direction-finding itself happened partly through building, not only before it.

## POV: When Does This Pattern Make Sense in the Real World?

**This pattern exists to solve a real N×M problem.** Without a shared standard, every combination of agent (Claude, Gemini, ChatGPT, and all their various deployments) and tool would need its own bespoke integration — its own credential format, its own consent flow, its own trust model. MCP solves that for tool integration generally; this authorization profile solves the same N×M problem specifically for *access control*. Without it, every agent-tool pairing would need its own custom auth handshake — exactly the kind of bespoke integration cost the base protocol exists to eliminate in the first place. Each agent implements the standard once, each tool implements the standard once, and access control "just works" between any pairing of the two.

**It's also specifically designed to keep credentials away from the model itself.** An agent never needs to see or hold a user's actual password or long-lived API key — it gets a scoped, time-limited bearer token through a flow it doesn't control the internals of. That distinction matters more with agentic systems than with traditional apps: a compromised or misbehaving agent leaking a scoped, short-lived, revocable token is a very different blast radius than one leaking a raw credential. This repo demonstrates the actual mechanics behind that claim, not just the assertion — PKCE protects the handshake itself from interception, audience binding means a token issued for one tool can't be replayed against another (the "confused deputy" problem), and scope enforcement means even a legitimately-issued token only grants a specific slice of capability.

**Where I think this pattern gets more expensive than it looks: the resource server's validation strategy becomes a real cost decision at scale, not a footnote.** The spec doesn't mandate a token format — opaque tokens validated via introspection, or self-contained JWTs validated locally. I hit this tradeoff directly in this project: I chose JWTs specifically to keep the *option* of skipping the introspection network hop, and weighed that explicitly against giving up live revocation checking. At real scale — tens of thousands of users generating tens of thousands of requests — that stops being an abstract tradeoff. Introspection-per-request against a metered SaaS authorization server (the choice we made here) means every protected request costs real money and adds real latency. A JWT-validated deployment scales for free, but only by accepting that a revoked token stays valid until it expires. That's a genuine architectural decision with a real cost curve behind it, not a detail to defer.

**It also solves client identity, not user identity — a different and harder problem.** Inside a large organization, "how do I implement OAuth" isn't actually the hard problem — it's "how do I know this *user* should be allowed to see or do this specific thing, and how do I track and audit that decision over time." That's a question about user identity and role governance, and it's worth being precise about what CIMD actually is: a way to establish *client* identity — which piece of software is calling, proven by domain ownership rather than pre-registration. It has nothing to do with which human is behind that client or what they should be allowed to do; this project keeps that distinction explicit by treating real user sign-in (Google SSO, an allow-list) as a separate, deliberately-deferred piece (Phase 8) rather than something CIMD was ever meant to cover. Swapping CIMD for something more gated, like Dynamic Client Registration behind an approval workflow, doesn't close that gap either — it only changes how *client applications* get vetted, which is the same category of problem CIMD solves, just with a human approving it instead of domain ownership proving it. Neither one touches the actual enterprise-hard question of which roles should hold which permissions; that's a layer of identity and access governance that sits entirely above and before any OAuth client-identification mechanism, CIMD included. That's not a flaw in CIMD — it was never scoped to solve that problem, and reaching for it (or its more gated cousins) as if it would be solving the wrong layer of the problem.

**Net POV:** this pattern earns its complexity when you actually have the N×M problem it was built for — many agents, many tools, no pre-existing trust relationship between them, and real security stakes if a credential leaks. It's a worse fit — or at least an incomplete one — inside a single organization with a small, known set of internal tools, where the harder problem is upstream (who gets which role) rather than downstream (how a token gets validated once issued), and where introspection-per-call economics can make the "just use a SaaS authorization server" default genuinely expensive at scale.
